- Web 侧只需改 HttpSseKernelClient.submit() 的 URL——它已经指向了新的 /turns
  端点（http-sse.ts:79）
- Server 侧的 kernel-routes.ts 已经实现了完整的 Session CRUD + Turn + Interrupt + Confirm
- formatEventMsg() 把 EventMsg 序列化为 SSE——已经实现

不用 axios，用一个集中的 fetch 封装

---

第一层：用不用 HTTP？—— 用，但要拆成两条通道

这是最关键的架构决策。Agent 通信天然是"请求-流式响应"模式，跟普通 REST CRUD 完全不同：

```
                  POST /v1/sessions/:id/turns    (发起任务)
    前端 ──────────────────────────────────────────────→ 内核
    前端 ←══════════════════════════════════════════════ 内核
                  SSE: token 流 / tool_call / progress    (接收流)

                  POST /v1/sessions/:id/interrupt (中断)
    前端 ──────────────────────────────────────────────→ 内核

                  POST /v1/sessions/:id/confirm   (审批)
    前端 ──────────────────────────────────────────────→ 内核
```

为什么不能塞进一个连接？

如果把控制命令（interrupt、confirm）也塞进 SSE 的同一条流里，就要在 SSE 的 data 字段里定义双向协议。这意味着你要自己实现帧类型、requestId 关联、超时重传——实质上是在 HTTP 上重新发明 WebSocket。反过来，"POST 发起 + SSE 接收"是 HTTP 原生的组合：

```
  ┌──────────┬──────────────────────────────────────┬──────────────────────────────────────┐
  │          │             POST（上行）             │             SSE（下行）              │
  ├──────────┼──────────────────────────────────────┼──────────────────────────────────────┤
  │ 谁发起   │ 前端                                 │ 内核                                 │
  ├──────────┼──────────────────────────────────────┼──────────────────────────────────────┤
  │ 连接模型 │ 短请求，即完即断                     │ 长连接，持续推送                     │
  ├──────────┼──────────────────────────────────────┼──────────────────────────────────────┤
  │ 代理兼容 │ 完全透明                             │ text/event-stream，标准 Content-Type │
  ├──────────┼──────────────────────────────────────┼──────────────────────────────────────┤
  │ 断线重连 │ 不需要（天然幂等或一次性的控制指令） │ EventSource 原生 Last-Event-ID       │
  └──────────┴──────────────────────────────────────┴──────────────────────────────────────┘
```

"POST 发起 + SSE 接收"比 WebSocket 更适合 coding-agent 场景——不是功能更多，而是协议栈更薄、出问题时更好排查。

---

第二层：SSE 的方向性 —— 下行 SSE + 上行 POST，用 sessionId 关联

SSE 是单向的（服务端 → 客户端）。文档明确指出两条数据流各走各的通道：

```
  下行 (内核 → 前端, SSE):
    Token("让我看看…")
    ToolCall({ name: "fs.read", args: { path: "a.ts" } })
    ToolProgress({ stream: "stdout", chunk: "..." })
    ToolResult({ ok: true, content: "..." })
    ToolConfirmRequired({ call: { ... } })
    TurnEnd({ reason: "completed" })

  上行 (前端 → 内核, 独立 HTTP POST):
    UserTurn      → POST /v1/sessions/:id/turns   (启动流)
    Interrupt     → POST /v1/sessions/:id/interrupt
    ConfirmTool   → POST /v1/sessions/:id/confirm

  关键约束：上行请求和下行 SSE 用 sessionId 关联。interrupt 和 confirm 是独立的 POST 请求——它们不经过 SSE 流，但服务器端的 InProcessKernelClient 根据 sessionId 找到对应的 AbortController 或 pendingConfirm，精确作用于正在跑的流。

  Liskin 当前代码已经做到了这个分离：

  // kernel-routes.ts — 已经是两条通道
  POST /v1/sessions/:id/turns      → kernel.submit()      → SSE 流
  POST /v1/sessions/:id/interrupt  → kernel.interrupt()   → 短请求
  POST /v1/sessions/:id/confirm    → kernel.confirmTool() → 短请求
```

第三层：axios 够不够？—— 控制类够，流式类不够

这是实现层面的核心坑。文档的结论是：

```
  ┌──────────────────────────────┬─────────────────────────┬──────────────────────────────┐
  │             通道             │         用什么          │            为什么            │
  ├──────────────────────────────┼─────────────────────────┼──────────────────────────────┤
  │ 控制类（interrupt、confirm、 │                         │ 拦截器统一加 header、错误码  │
  │ session CRUD、provider       │ axios ✅                │ 、超时重试——这是 axios       │
  │ CRUD）                       │                         │ 的强项                       │
  ├──────────────────────────────┼─────────────────────────┼──────────────────────────────┤
  │                              │ fetch + ReadableStream  │ 浏览器端 axios 底层是        │
  │ 流式类（submit → SSE 消费）  │ 或 @microsoft/fetch-eve │ XMLHttpRequest，不支持真正的 │
  │                              │ nt-source ✅            │ 流式读取                     │
  └──────────────────────────────┴─────────────────────────┴──────────────────────────────┘
```

axios 在浏览器端为什么不能做流式？

axios 浏览器端:
XMLHttpRequest.onprogress → 累积的 responseText → 自己切分 \n\n
问题: 粘包、丢包、无法逐 chunk 处理

原生 EventSource:
只支持 GET、不能带 body、不能自定义 header(没法传 Authorization)
问题: 发不了 POST、带不了 token

@microsoft/fetch-event-source:
底层是 fetch + ReadableStream → 真正的逐 chunk 消费
支持 POST、自定义 header、自动重连、onmessage/onerror 回调
✅ 专门为 LLM 流式场景设计

落地结构

文档给出的文件结构：

```
  web/src/kernel/
    http.ts        // axios 实例 + 拦截器（控制类：session CRUD、interrupt、confirm、provider CRUD）
    stream.ts      // fetch-event-source 封装（流式类：submit → SSE 消费）
    client.ts      // HttpSseKernelClient：实现 KernelClient 接口，组合 http.ts + stream.ts

  client.ts 是关键——它实现 KernelClient 接口（和 InProcessKernelClient、未来的 JsonRpcKernelClient 同签名），把 axios 和 fetch-event-source 的差异完全封在内部。上层 UI 只依赖 KernelClient 接口，不感知底层是 HTTP 还是 SSE。
```

```
  ┌────────────────────────────────────────────┐
  │  Chat.tsx / useSessionStore                 │  ← 只 import KernelClient 接口
  │    kernelClient.submit(...)                │
  │    kernelClient.interrupt(...)             │
  └──────────────┬─────────────────────────────┘
                 │ 依赖接口
  ┌──────────────▼─────────────────────────────┐
```
