核心执行（读写代码、跑命令）通常应在本地完成。  
 那还要 FaaS + AGW，主要是为了“服务化能力”，不是替代本地执行。

---

一句话结论

- 本地 Agent：负责“重活”与环境相关动作（代码操作、git、构建、测试）。
- AGW/FaaS：负责“统一入口、治理、弹性、跨端共享能力”。

---

为什么还需要 AGW

AGW（API Gateway）解决的是平台层问题：

1. 统一入口
   - Web/IDE/CLI 都走一套 API，不是每个端直连每个后端服务。

2. 鉴权与权限控制
   - 登录态、token 校验、租户隔离、限流、审计统一处理。

3. 协议适配
   - 前端 HTTP 请求 -> 内部 RPC/服务调用，屏蔽后端细节。

4. 灰度与可观测
   - 路由策略、埋点、错误码、熔断、降级都在网关做。

▎ 没 AGW 的话，本地 agent 之外的端到端能力会变得很散、难管、难审计。

---

为什么还需要 FaaS

FaaS 不是为了跑本地 coding 动作，而是为了这些场景：

1. 弹性处理短任务/突发流量
   - 例如转发、格式转换、轻量编排、hook 回调。

2. 流式代理
   - 你们的 /forward/stream

这种能力很典型：把下游流式输出稳定透传给上游客户端。  
 3. 隔离与快速迭代

- 某些逻辑可独立发布，不影响主服务（AGW/核心后端）。

4.  成本与运维

- 低频能力放 FaaS，按需扩缩，不常驻占资源。

---

结合 coding agent 的典型分工

- 本地执行（必须本地）
  - 文件读写、代码修改、运行测试、git 操作。
- 服务端（AGW/FaaS）
  - 会话管理、鉴权、配额、模型路由、事件上报、流式中转、跨端同步。

---

什么时候可以不要 FaaS？

如果你是纯本地单机工具（无多用户、无统一网关、无云端协作），可以先不要
。  
 但一旦要支持 Web/多人/组织级治理，AGW 基本是刚需，FaaS  
 则是高性价比补充层。
