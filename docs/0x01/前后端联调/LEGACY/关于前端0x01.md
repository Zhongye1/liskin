长期迭代路由几乎肯定要改，而且不止路由。 但要分清哪些是现在就得动、哪些是等功能长出来再动，别提前过度设计。
先看现状：路由现在是 "占位级" 的
当前 router.tsx 只有两条路由 ——/(Home) 和 /chat(Chat), 且 App.tsx 还停留在 "liskin code /web init" 的脚手架状态，和我们上一轮讨论的两栏时间线 UI 完全对不上。更关键的是，会话状态全在一个 Zustand 全局 store 里，activeSessionId 是 store 字段而不是 URL 参数。这意味着现在切换会话不会改变 URL, 刷新页面、分享链接、浏览器前进后退都无法定位到具体某个会话。这正是长期迭代第一个会撞到的墙。
路由该怎么改：把会话 ID 提到 URL 里
参照你给的 Claude Code 截图，它的形态是 "左栏会话列表 + 右栏某个具体会话", 这种结构的标准做法是让会话 ID 进入路由:
tsx
createBrowserRouter([
{
path: '/',
element: <App />, // 两栏 shell:Sidebar 常驻 + Outlet
children: [
{ index: true, element: <EmptyState /> }, // 未选会话时的引导页
{ path: 'sessions/:sessionId', element: <Conversation /> },
{ path: 'settings', element: <ProviderSettings /> },
],
},
]);
这样做的收益是实打实的：URL 成为单一真相源，activeSessionId 从 store 字段退化成 useParams() 读出来的值，刷新能恢复现场、链接可分享、前进后退天然可用。代价是 session-store.ts 里 selectSession 的语义要从 "setState" 改成 "navigate",store 不再持有 activeSessionId。这是一次有意义的架构调整，不是表面换皮 —— 本质是把 "哪个会话是当前会话" 这个状态的归属从内存搬到 URL。这件事越早做越好，因为越往后会话相关逻辑越多，迁移成本越高。
布局这块用 react-router 的嵌套路由 + <Outlet> 很自然:App 渲染常驻的 Sidebar 和右侧 Outlet,Conversation 通过 Outlet 注入，Sidebar 不随路由卸载，符合 "列表常驻、右栏切换" 的体验。
但路由只是其中一环，长期迭代要一起规划的有这几样
状态管理的边界要重新划。 现在所有东西都堆在一个 session-store.ts, 长期会膨胀成上帝对象。合理的切分是：服务端数据 (会话列表、历史消息、provider 列表) 交给 SWR——package.json 里 SWR 已经装了但似乎还没真正用起来，它能白送你缓存、重验证、乐观更新；真正的本地 UI 状态 (草稿、流式 buffer、确认弹窗) 才留在 Zustand; 而 "当前会话是谁" 交给 URL。三者各管一段，store 会瘦下来。
依赖要做减法。 现在 web 同时装了 axios、swr、ahooks、usehooks-ts 四个工具库，而实际网络请求走的是 http-sse.ts 里的原生 fetch,axios 大概率没用上；Readme 的 "接下来工作" 里也明确写了要 "清掉未使用的依赖 (ahooks、axios、swr、usehooks-ts)" 和 "react-markdown/highlight.js 改 dynamic import 消除 614KB chunk"。长期迭代前先把依赖收敛清楚，否则 bundle 只会越滚越大。这里有个张力：上面我建议用 SWR、Readme 却想砍 SWR—— 你需要先定一个决策，要么 "SWR 作为服务端状态层正式启用", 要么 "统一用 fetch + 自写 hook", 二选一，不要两者都留着半用不用。
代码分割要随路由一起上。 一旦路由拆成 sessions/settings 多页，就用 lazy() + Suspense 按路由切 chunk, 把 monaco、mermaid、react-markdown 这些重组件挪到用到它们的路由里懒加载，首屏只载 shell。路由重构和 code splitting 是天然一起做的，分开做等于做两遍。
错误与加载边界要补。 现在 main.tsx 只有一个 RouterProvider, 没有 ErrorBoundary、没有路由级 errorElement、没有 loading 态。长期迭代里 SSE 断流、会话 404、provider 没配这些都要有兜底页，react-router 的 errorElement 和 loader/useNavigation 正好覆盖，值得在重构路由时一并搭好骨架。
怎么排优先级
我会这样分批，避免一次动太多:
第一批是地基, 和上一轮的 UI 重构一起做 —— 把 App.tsx 改成两栏 shell、路由加上 sessions/:sessionId、activeSessionId 迁到 URL、补 ErrorBoundary 和路由 errorElement。这批是后面一切的前提，必须先做。
第二批是状态与依赖收敛—— 定 SWR 还是 fetch、砍掉确认没用的库、瘦身 store。这批不阻塞功能但越拖越贵。
第三批是按需性能—— 路由级 lazy、重组件 dynamic import、长列表虚拟滚动 (@tanstack/react-virtual 也已经装了，时间线长了之后用得上)。这批等真有性能问题或会话历史变长再做，提前做是浪费。
至于路由库本身要不要换 (比如上 TanStack Router 拿类型安全路由，或者上 Next), 我的判断是短期不必:react-router v6 的 data router 已经够用，贸然换框架的收益撑不起迁移成本，等到真有 SSR/SEO 或文件路由的硬需求再考虑。
需要的话，我可以把第一批直接落到 liskin 的 web 目录里 —— 重写路由、改 store 让会话 ID 走 URL、搭好两栏 shell 和错误边界，然后跑通 tsc 和现有 17 个 web 测试
