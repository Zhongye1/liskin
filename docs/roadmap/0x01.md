## 四、关键架构决策（M0 时就要钉死）

### 1. 端口模式（Ports & Adapters）保证内核独立

core 只依赖接口：

    // packages/core/src/ports/llm-port.ts
    export interface LLMPort {
      chatStream(req: ChatRequest): AsyncIterable<LLMEvent>;
    }
    // 没有任何 import openai / anthropic

llm 包实现这个接口：

    // packages/llm/src/openai.ts
    import type { LLMPort } from '@liskin/core/ports';
    export class OpenAIProvider implements LLMPort { ... }

这样未来加 Anthropic / 后端网关，只是新增一个 LLMPort 实现，core 零改动。

### 2. Tool 调用走端口，不让 core 直接 import tools

同样 core 定义 ToolPort，tools 实现它。

### 3. Harness 接口先留，Phase 0 用 noop 实现

core/harness/noop.ts 实现一个什么都不做的 harness，签名是真的 — 这样 Phase 1 真正写  
harness 时，core 代码零改动。

### 4. 依赖红线用工具校验（不靠自觉）

M0 加 dependency-cruiser 或简单的 madge 检查脚本，CI 卡住反向依赖：

    # .dependency-cruiser.cjs
    forbidden:
      - name: 'core-must-not-depend-on-llm-tools-server'
        from: { path: '^packages/core' }
        to:   { path: '^packages/(llm|tools|server|web)' }

# 二、Port 是干什么的

一句话：Port 是「内核与外界的电源插座」— 接口在内，实现在外。

## 不用 Port 的世界（坏例子）

    // 后果：
    // - 想换 Anthropic 要改内核
    // - 想换 IndexedDB（浏览器跑）要改内核
    // - 想跑单测要 mock 整个 OpenAI SDK，巨痛

## 用 Port 的世界（现在的实现）

    // packages/core/src/ports/llm-port.ts
    export interface LLMPort {
      chatStream(req: ChatRequest): AsyncIterable<LLMEvent>;
    }
    // ↑ 内核只知道「有个东西能流式聊天」，不知道是 OpenAI 还是 Anthropic

    // packages/core/src/agent/loop.ts
    export async function* runAgent(opts: { llm: LLMPort; tools: ToolPort; ... }) {
      for await (const ev of opts.llm.chatStream({ messages, tools })) {
        // 内核只跟接口打交道
      }
    }

• packages/llm 写一个类 OpenAIProvider implements LLMPort （M2 要做）  
• 想加 Anthropic？再写一个 AnthropicProvider implements LLMPort  
• 内核代码零改动

## 关键实现亮点

### 1. Tool Call 增量拼接（这是 OpenAI 协议最容易出错的地方）

OpenAI 的流式 tool_call 把 arguments（JSON 字符串）分片下发，按 index  
标识槽位。stream.ts 用 Map<index, PendingToolCall> 跟踪每个槽位累积，直到 finish_reason:
'tool_calls' | 'stop' 触发一次性 flush + JSON.parse。

### 2. AbortSignal 三段防御

• chatStream 入口检查  
• 每个 chunk 循环开头检查  
• finish_reason flush 前再检查

### 3. 错误归一化覆盖 3 类

• APIError → 按 status 映射 code: '401' | '429' | ...  
• 普通 Error → code: 'unknown'  
• 非 Error → String(err)

### 4. 容错策略

非法 JSON 的 arguments 不抛错，包成 { \_raw: <string> } 让上层 tool 校验时再决定 — 与 M1
状态机的 tool_result.ok=false 路径自然衔接。

## 关键实现亮点

### 1. preflight 签名 = (call, ctx)

拿到完整 ToolCall（含真实 id），applyConfirmPolicy(call, policy) 抛  
ConfirmRequiredError(call) 时携带原始 callId — 这样上层 runAgent 用  
confirmedCallIds 做精确匹配。

### 2. confirmedCallId 命中后仍跑 preflight

但用 confirmPolicy: 'auto' 替换原策略 — 路径白名单 /  
危险命令拦截这种非确认类拒绝仍生效，只跳过 ask  
弹窗。这是一个细致但关键的设计：用户确认了一次，不代表后续可以绕过 sandbox。

### 3. 路径白名单防 path traversal

用 path.resolve() 后判断 abs === allowed || abs.startsWith(allowed + sep)，防止  
../../etc/passwd 这种穿越。

### 4. 危险命令 regex 取舍

9 条默认黑名单覆盖最危险的：rm -rf / / rm -rf ~ / curl|sh / mkfs.* / dd  
if=*of=/dev/\* / 写 .ssh / 写 .env。原则：宁可漏过偏门、不误伤正常。echo hello >  
/tmp/x.txt 等正常重定向不拦。

### 5. shell 用 sh -c

保留 shell 语义（管道 / 重定向 / 引号），危险命令由 command-policy 拦截。execa@9  
的 cancelSignal（不是 v8 的 signal）传入 AbortSignal。

### 6. 非零退出归一为 ok:true

shell.exec 退出码非零时不抛错，返回 ok:true + content 含 exit code  
N。让模型自己根据 stdout 决策，而不是直接当失败处理。

## 为什么删 llmGateway.ts 和 types/llm.ts

• web/src/services/llmGateway.ts 是 M0 之前给「Web 直连 Go faasrouter  
网关」用的：用 axios 把请求打到 /api/gateway/model，请求体里塞 target_url + Bearer
token，由 Go 后端 forward 到上游 super-relay。M4 之后通路彻底变成「Web → 本地  
agent → OpenAIProvider」，API key 由 agent 服务端持有，前端不再持有  
token，也不该自己拼 OpenAI 协议字段。接管它的是 agentClient.ts，消费的是高阶  
AgentEvent（token / tool_call / tool_result / tool_confirm_required / done /  
error）而不是裸的 OpenAI  
delta。两套通信完全不重叠，留着只会让前端有两条路径外加一份过期的 axios  
流式拉取代码。  
• web/src/types/llm.ts 是它的伴生类型（LLMRole / LLMMessage /  
StreamChatParams），整份文件只被 llmGateway.ts 和改写前的 useLLMChat.ts 用过，删除
gateway 后即成死代码。Phase 0 的权威消息/事件类型已经在内核包 @liskin/core（Msg /
AgentEvent / ToolCall / ToolResult）里定义好，web 端通过 agentClient.ts  
引用一份镜像即可，没必要再维护一份职责重叠的 LLM 类型。
